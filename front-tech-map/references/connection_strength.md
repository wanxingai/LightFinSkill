# 连接强度分层体系

> 本文档定义了技术 mapping 中"什么关系值得画线"的判定标准。

## 三级连接强度

| 级别 | 连线策略 | 说明 |
|------|---------|------|
| **Tier 1 — 直接传承** | ✅ 必须画线 | 有明确的指导、共建、创办关系 |
| **Tier 2 — 强关联** | ✅ 画线，标注为弱连线 | 有合著奠基论文或代码派生关系 |
| **Tier 3 — 弱关联** | ❌ 不画线 | 仅有引用、同校、同会议等间接关系 |

---

## Tier 1：直接传承（必须画线）

满足以下**任一**即为 Tier 1：

| 关系类型 | 判定条件 | Graphviz 边色 | Label |
|---------|---------|--------------|-------|
| PhD 师承 | A 是 B 的博士/硕士导师 | `#5b9bd5` 蓝 | "PhD advisor" |
| Postdoc 师承 | A 是 B 博后阶段的 PI | `#5b9bd5` 蓝 | "postdoc advisor" |
| 联合创办 | A 与 B 共同创办了公司 C | `#27ae60` 绿 | "co-founded" |
| 同一实验室核心成员 | 同时期在同一 PI 的实验室工作（需有共事证据） | `#5b9bd5` 蓝 | "lab colleague" |
| 同一开源项目核心 Maintainer | 均为项目 top contributors，有 merge 权限 | `#e69138` 橙 | "co-maintainer" |

**证据要求**：至少 2 个独立信息源确认（如 OpenAlex + 大学官网 / LinkedIn + GitHub）。

---

## Tier 2：强关联（画线但标注）

满足以下**任一**即为 Tier 2：

| 关系类型 | 判定条件 | Graphviz 边色 | Label |
|---------|---------|--------------|-------|
| 合著奠基论文 | 该论文开创了一个技术分支（被引 1000+ 或领域公认奠基），双方为共同一作或通讯作者 | `#cc4125` 红 | "co-authored [论文简称]" |
| GitHub 重要 Fork | 从项目 A fork 后独立发展为有影响力的新项目 B（star 数达到原项目 10%+） | `#e69138` 橙 | "forked from" |
| 技术借鉴/独立发展 | A 和 B 在不同地方独立发展出类似技术，但有公开证据表明技术方案受对方启发 | `#e69138` 橙 | "独立发展 [技术名]" |

### 论文关系判定流程图

```
A 的论文引用了 B 的论文
  ├── A 和 B 是共同作者吗？
  │   ├── YES → 是奠基级论文吗（被引1000+ 或开创技术分支）？
  │   │   ├── YES → 双方是一作/通讯吗？
  │   │   │   ├── YES → ✅ Tier 2（合著奠基论文）
  │   │   │   └── NO → 检查是否为师生关系 → 若是 → ✅ Tier 1
  │   │   └── NO → 是师生关系吗？
  │   │       ├── YES → ✅ Tier 1（导师-学生论文）
  │   │       └── NO → ❌ Tier 3（普通合著，不画线）
  │   └── NO → ❌ Tier 3（纯引用关系，不画线）
```

> ⚠️ **核心原则**：引用 ≠ 传承。一个领域的高引论文会被所有人引用，这不构成任何连接关系。

---

## Tier 3：弱关联（不画线）

以下关系**不画线**，也不在图谱中体现：

| 关系类型 | 为什么不画 |
|---------|----------|
| 普通论文引用 | 垂直领域高引论文被所有人引用，无法体现真实传承 |
| 同一大学不同系 | "都在 MIT" 不代表有合作，太泛 |
| 同一学术会议演讲/同一 workshop | 参加同一个会不代表有关系 |
| 非核心 PR 贡献者 | 提了几个 PR 不算有传承关系 |
| 同一投资方投资 | 投资关系不是技术传承 |
| 同一行业协会/标准组织成员 | 太泛，不画线 |

## 边界案例处理

### 案例 1：高引综述论文
某综述论文被引 5000+，A 和 B 都引了它。
→ ❌ Tier 3。综述论文是公共知识，不构成连接。

### 案例 2：A 的博士论文大量引用 B 的工作
但 A 和 B 不是师生关系，也没有合著。
→ ❌ Tier 3。引用只说明 A 知道 B 的工作。

### 案例 3：A 在 B 的实验室做了 3 个月的访问学者
→ 需要判断：是否有合著论文？是否有公开证据表明 B 对 A 的研究方向有实质影响？
- 有合著 → ✅ Tier 1（短期但有实质合作）
- 无合著 → ❌ Tier 3

### 案例 4：核心开发者 A 从大厂离职后创办了竞品公司
→ ✅ Tier 1（从属关系），边 label 标为 "前核心成员" 或 "spun off from"

### 案例 5：A 和 B 都是某大厂 AI Lab 的核心成员，后分别创业
→ ✅ Tier 1（同一实验室核心成员），边 label 标为 "lab colleague at [大厂名]"
→ 类似于学术实验室的"同门"关系，但载体是大厂 Lab

### 案例 6：A 的开源项目大量使用了 B 的框架，但没有 fork 关系
→ 需要判断：是否为核心技术依赖（如 vLLM 依赖 PyTorch）？
- 核心创新建立在 B 之上 + A 曾在 B 的团队工作 → ✅ Tier 1
- 仅作为通用工具使用（如几乎所有项目都用 PyTorch）→ ❌ Tier 3

### 案例 7：Wikidata 查询确认了 A 是 B 的博士生
→ ✅ Tier 1（PhD 师承），Wikidata P184 属性为高可信第三方源
→ 置信度：85%（如有额外搜索引擎验证则提升至 95%）
